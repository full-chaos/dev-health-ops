# Write and review documentation

- Lead with the reader outcome and prerequisites.
- Use current product labels and routes only after source verification.
- Separate steps, expected result, failure path, exact reference, and next task.
- Distinguish observed, derived, model-assisted, estimated, unavailable, incomplete, stale, and unsupported states.
- Use visuals only when they reduce reader effort; include text equivalents, source revision, sanitization, owner, and review trigger.
- Link to canonical facts rather than copying large tables.
- Remove internal project language and implementation history.

Review source accuracy, information architecture, content, accessibility, security, and operational safety according to page risk.
